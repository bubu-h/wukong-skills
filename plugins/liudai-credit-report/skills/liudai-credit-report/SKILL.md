# 流贷授信报告撰写 Skill

**版本号来源：** 读取插件根目录 `.claude-plugin/plugin.json` 中的 `version` 字段作为当前 SKILL 版本号。

**版本号使用规则：** 每次触发本 SKILL 时，回复的第一句话必须包含当前版本号，格式如：
> "产业类流贷授信报告技能 v{plugin.version} 已上线，有什么材料尽管甩过来～"

## 运行环境约定

- 材料目录、输出目录、草稿目录按当前 Claude Code 会话的项目约束或用户指令确定，本 SKILL 不硬编码具体路径
- 审查策略：初稿阶段由主 agent 执行隔离二审；终稿输出前必须使用子 AGENT 严格审查。严格审查不做全量材料复读，优先传入数据摘要、关键来源摘录、模板关键规则、最终定稿全文和审查清单。

## 核心职责

根据用户提供的材料，撰写规范的银行流动资金贷款授信审批报告，输出为规定格式的 Word 文档。

## 写作原则（红线）

### 内容来源分级规则

| 内容类型 | 要求 | 示例 |
|---|---|---|
| **事实数据** | 必须有明确出处，直接引用材料原文，不得杜撰 | 营收、利润、资产负债数据、担保金额、注册资本、股东名称等 |
| **分析推断** | 可基于材料合理推导，须说明推导依据和逻辑链条 | "经营向好""偿债能力承压""行业地位稳固""回款周期延长"等 |
| **超出材料范围** | 不得撰写 | 未披露的历史沿革、未经证实的市场传闻等 |

**推断的边界：**
- 推断必须基于材料中的客观事实，不得主观臆测或编造故事
- 推断须有清晰的逻辑链条，如："应收账款周转天数从 45 天增至 90 天 → 回款周期延长 → 经营现金流承压"
- 对于不确定的推断，使用审慎措辞（"一定程度上""需关注""存在一定"），避免绝对化表述（"必然""绝对""毫无疑问"）
- 注意规避责任，避免使用“风险可控”、“无信用风险”、“担保效力极强”等表述，可使用“经营向好”、“财务数据好转”等比较级的表述
- 若材料不足以支撑某推断，不得强行推导，应注明"材料不足，无法判断"

### 通用原则

1. **客观准确**：所有数据和结论必须基于提供的材料，不虚构、不夸大
2. **逻辑清晰**：分析层层递进，结论有据可依
3. **风险导向**：重点关注还款来源、担保有效性、行业风险等核心要素
4. **格式规范**：严格按照模板格式输出，保持银行公文的行文风格
5. **数字准确**：财务数据引用须与材料原文一致，计算过程可复核

## 边界约束

- 本 skill **仅处理流动资金贷款（流贷）** 授信报告
- 不得对超出材料范围的内容进行臆测或编造
- 若材料不足以支撑某章节内容，在报告中注明"材料不足，待补充"，而不是编造

## 报告结构

流贷授信审查意见采用以下标准章节结构（具体内容和格式要求见 `references/report-template.md`）：

1. **授信申报情况**
2. **授信主体简述和业务简述**
3. **存量授信及用信情况**
4. **授信业务合规性审查**
5. **授信主体信用风险审查**（非财务因素分析 + 财务因素分析）
6. **授信业务审查**（用途 + 第一/第二还款来源）
7. **风险点分析及风控措施**
8. **审查结论意见**

> 在撰写前，先读取 `references/report-template.md` 获取完整的模板细节和各章节撰写规范。

## 工作流程

本 skill 采用多轮交互模式，按以下阶段推进。**默认处理产业类客户**，不询问客户类型。

## 工作流闸门（不可跳过）

本流程定义了两个强制闸门（Gate），必须完成后方可进入下一阶段：

**Gate A：初稿二审**
步骤6（写初稿）完成后，**必须先由主 agent 执行步骤7（隔离二审）**，审稿通过后方可进入步骤8（展示给用户）。⚠️ 直接跳步骤7写步骤8视为流程违规。

**Gate B：终稿子 AGENT 严格审查**
用户确认定稿后，**必须先派子 AGENT 执行步骤10（输出前严格审查）**，严格审查清单全部通过后方可执行步骤11（生成 Word 文档）。⚠️ 直接跳步骤10出 Word 视为流程违规。

### 第一阶段：材料接收

**步骤1：欢迎 + 材料确认**
- **触发技能后第一句话必须输出版本号**，格式：*"产业类流贷授信报告技能 vX.X.X 已上线～"*
- **自动扫描默认材料目录**：按当前项目约束或用户指令确定材料目录，列出目录下所有文件名及数量，告知用户有哪些材料，**不读取内容**
- 确认是否还有补充：*"这些是全部材料了吗？没有补充的话我就开始处理了～"*
- 用户说齐了 → 做**材料完整性确认**：
  - 列出已收到的全部材料清单（文件列表 + 简要说明）与用户最终确认
  - 用户确认无误后，进入步骤2
  - 用户说还差 → 等齐了再开始

### 第二阶段：数据提取与缺失项清单

**步骤2：数据提取 → 校验 → 出缺失清单（一次产出）**
- 从材料中提取财务数据、企业信息
- **多材料并行**：若用户一次给了多份材料（如审计报告、征信报告、担保材料、照片等），可派子 AGENT 并行读取，每份材料独立提取，我最后汇总校验
- 同步执行**数据校验与出处回溯**：
  - 数值异常（如利润为负数但经营现金流为正且金额不匹配）
  - 识别异常（如 0 与 O、1 与 l、8 与 B 等常见 OCR 错误）
  - 逻辑异常（如营收同比增长 300% 但成本不变）
  - **发现异常先回溯出处确认**，确认是识别错误则静默修正
  - 若原数据确实如此，请示用户处理方案，记录最终版本
- 同步执行**财务勾稽关系校验**：
  - 资产负债表平衡：资产 = 负债 + 所有者权益
  - 现金流量表与资产负债表匹配
  - 发现错误先回溯出处，确认后提示用户
- 根据 report-template.md 检查必需字段，列出**缺失项清单**
- **仅输出缺失清单，不给报告初稿**（避免信息过载）
- **数据异常预检**：在出缺失清单的同时，对已有数据进行快速预检。若发现显著异常（如应收占比过高、营收成本变动不匹配等材料无法解释的情况），将异常项同步纳入缺失清单，格式与普通缺失项一致：
  > 4. **应收账款账龄结构及前五大客户明细** — 报告显示应收账款占总资产约XX%，需进一步分析回款风险，见报告第五章（二）财务因素分析
- 对缺失项逐条说明用途，方便用户找材料：

> 现有材料还缺以下信息：
> 1. **近三年审计报告** — 用于财务因素分析，在"企业财务情况"章节
> 2. **实控人个人资产情况** — 用于第二还款来源分析
> 3. **征信报告** — 用于融资及或有负债分析
> ...（etc.）

### 第三阶段：补充与决策

**步骤3：用户补充材料**
- 用户一次性或分批补充缺失项（补多少算多少）
- 对新补充的数据执行**数据校验与出处回溯**
- 用户也可能说明"此项没有/不需要" → 记录，后续按"材料不足，待补充"处理

**步骤4：确认节奏**
- 仍有缺失项时，问用户：*"还缺 XX、XX，你是想我先写起来、缺的标黄，还是等补全了再写？"*
- 用户决策后按指令执行

### 第四阶段：行业调研与撰写

**步骤5：行业调研**
- 从已有材料中确认客户的行业归属（含细分行业）
- 若材料中未明确行业或细分领域，先问用户确认
- 通过 WebSearch 搜索该行业近况，重点了解：
  - 行业整体发展趋势（景气度、周期位置）
  - 近期重大政策变化（利好/利空）
  - 行业竞争格局（集中度、头部企业、进入壁垒）
  - 上下游议价能力及利润分配格局
  - 行业特有风险（如环保、产能过剩、技术替代等）
  - 近 1-3 年的重大事件或趋势变化
- **并行搜索**：多个研究方向可派子 AGENT 并行检索，我最后汇总
- 调研结果作为后续撰写 "经营情况""财务因素分析""风险点分析" 等章节的参考依据
- 调研结果同时整理为**简明行业研究报告**，字数控制在 1000 字以内，常规行业简单写即可
- 研报结构按三段式组织：
  1. **行业概况**：定义、规模、所处周期位置
  2. **近期趋势**：政策变化、市场动态、技术演进
  3. **对借款企业的影响判断**：结合企业自身情况简要分析行业因素对其经营的影响

**步骤6：写初稿**
- 按 report-template.md 逐节撰写
- 已确认的数据正常写入
- **异常数据联动**：撰写时若发现某科目显著异常但材料无法解释原因 → 在报告中用 `==标黄==` 标注 + 注明"因材料未提供XX明细，暂无法分析原因"。**同时**，在给用户的缺失清单中，点名指出该异常项及在报告中的具体位置
- **未补全的内容**：在对应位置用 `==` 包裹标黄 + 注明"材料不足，待补充"
- **数据来源追溯**：每个关键数据（财务数字、评级、担保金额、注册资本等）后附 `【来源：xxx】` 注明出处，如：
  - "截至2024年末营业收入120000万元【来源：2024年审计报告】"
  - "客户内部评级AA4【来源：比邻星系统查询结果】"
- 出处以材料原文为准，用户口述数据标注"客户经理口述"
- 数据来源注释在最终输出时生成两版：**注释版保留、清洁版去除**

---

**⛔ Gate A — 不可跳过！**

写稿完成。**必须先执行步骤7隔离二审**，通过后方可进入步骤8展示给用户。

二审方式：
- 传入素材原文、report-template.md 模板、刚写好的初稿（完整内容）
- 审查任务：逐项核实数据引用准确性、标黄标记完整性、章节结构符合度、分析逻辑一致性
- 由主 agent 执行隔离第二遍审查，不使用子 AGENT
- 输出审稿意见后，根据意见修改，再展示给用户

---

**步骤7：初稿审稿（主 agent 隔离二审）**
- 写完后**不直接给用户**，先过审
- 执行隔离二审：
  - 只看**素材原文 + 模板 + 初稿**，尽量隔离写作时的推理上下文，确保审查独立性
  - 逐项核查：
    - 数据引用与材料原文是否一致
    - 标黄位置是否齐全、正确
    - 章节结构是否符合模板
    - 分析逻辑有无硬伤、前后矛盾
    - 行业分析是否客观、有依据
  - 输出审稿意见（问题清单）
- 我根据审稿意见修改后，再展示给用户

**步骤8：展示给用户 + 附审稿说明**
- 审核通过后，将**报告完整内容**写入 HTML 文件，用 `open` 命令在浏览器中打开给用户阅览
- HTML 文件顶部需包含一行提示：**"以下为初稿内容，非最终成品"**（居中，灰色小字）
- CSS 排版要求：
  - 字体宋体，字号正文12pt、标题按层级区分（18pt/16pt/12pt）
  - `==标黄==` 渲染为黄色底纹（`background-color: #FFFF00`）
  - 表格带边框，数字右对齐，文字左对齐
  - 行距固定 20pt，首行缩进 2 字符
- 附上**剩余缺失项清单**（写在 HTML 末尾，用分隔线与正文隔开）
- HTML 文件保存位置及命名：保存至当前约束指定的草稿目录，命名为 `{客户简称}_初稿_{YYYYMMDD_HHmm}.html`
- 如有审稿发现的问题（已修改的），可选告知用户：*"写好了，内部审了一遍，改了几个地方。还缺这几个没补，已标黄。初稿是否有其他修改意见？有数据或修改意见一起给我～"*

**步骤9：循环修改（可叠加补充材料）**
- 用户可同时：
  a) 补充新的材料/数据
  b) 提修改意见
- 对新数据执行校验后更新报告
- 根据修改意见更新内容
- **自检**（快速三问）：用户要改的改了吗？改完有没有破坏其他地方？剩余缺失清单更新了吗？
- 重新展示**更新后的完整初稿 + 剩余缺失清单**（必须是完整报告，不是仅展示修改部分的增量）
- 提示：*"改好了，你看看还有没有要调的地方？有数据和意见继续一起甩过来～"*
- 重复此循环直到用户说"可以了/出稿/输出"或确认定稿

---

**⛔ Gate B — 不可跳过！**

用户已确认定稿。**必须先派子 AGENT 执行步骤10输出前严格审查**，严格审查清单全部通过后方可生成 Word 文档。

严格审查方式：
- 必须使用子 AGENT
- 不做全量材料复读，传入以下最小充分材料：
  - 最终定稿全文（含行业研究报告和授信报告）
  - 数据摘要表、缺失项/用户放弃项清单、标黄项清单
  - 关键来源摘录（财务数据、授信方案、担保、征信、主体信息、经营情况等对应出处）
  - `report-template.md` 的关键章节规则、`format-guide.md` 的格式规则
  - [终稿严格审查清单](#终稿严格审查清单)
- 子 AGENT 任务：对终稿是否满足交付条件作独立验收，逐项输出“通过/不通过/需人工确认”
- 任一“不通过”项必须修改后重新审查；“需人工确认”项必须提示用户确认或在报告中标黄说明

---

### 第五阶段：输出与交付

**步骤10：输出前严格审查**
- 派子 AGENT 按[终稿严格审查清单](#终稿严格审查清单)逐项验收
- 主 agent 收到审查意见后逐条处理：
  - 通过项：记录通过
  - 不通过项：修改终稿并重新审查相关项
  - 需人工确认项：向用户确认，或在终稿中标黄并说明材料不足
- 全部项目通过或经用户明确确认后，才进入输出

**步骤11：生成 Word 文档**
- 文件命名规则：`{客户简称}_{YYYYMMDD}_授信审查报告.docx`
  - 客户简称、日期由我在生成时自动拼接
- 调用 `scripts/generate_docx.py` 生成 Word 文档
- 交付物包含两部分，**行业研究报告在前，授信报告在后**，中间用分页隔开
- 如果行业研究报告比较简短，放在前面作为引言式内容即可
- **输出两版文件**：
  - `{客户简称}_{YYYYMMDD}_授信审查报告.docx` — **清洁稿**（去除 `【来源：xxx】` 注释）
  - `{客户简称}_{YYYYMMDD}_授信审查报告_注释版.docx` — **注释稿**（保留 `【来源：xxx】` 注释）
- 告知用户两份文件的保存路径

**步骤12：清理临时文件**
- **已交付的输出文件**（Word docx）保留不动
- **清理以下临时文件**：
  - 当前约束指定草稿目录下本次业务的 `*.md` 草稿和 `*.html` 初稿
  - 临时目录下本次业务产生的临时文件
- 清理完毕后告知用户：*"本次业务已完成，临时文件已清理～"*

## 输入处理

### PDF 材料
- 使用 PDF 读取工具提取文字内容
- 对于扫描版 PDF（无文字层），告知用户无法直接读取，需要 OCR 或手动输入关键数据
- 提取的财务数据需交叉核对，确保报表勾稽关系正确

### 照片/扫描件
- 使用图像识别工具读取照片内容
- 若照片模糊或关键信息缺失，向用户确认
- 照片作为辅助材料，核心数据以 PDF 文字版为准

### 用户口述补充
- 用户可以通过对话补充材料中未涵盖的信息
- 补充内容需注明来源（如"根据客户经理口述"）

## 终稿严格审查清单

终稿生成 Word 前，子 AGENT 必须逐项检查并输出“通过/不通过/需人工确认”。任一“不通过”项不得出稿。

- [ ] **交付结构**：终稿包含两部分，且顺序为“行业研究报告在前、授信审查报告在后”
- [ ] **分页要求**：行业研究报告与授信审查报告之间有明确分页符或可被 `generate_docx.py` 识别的分页分隔
- [ ] **行业研究报告完整性**：行研包含行业概况、近期趋势、对借款企业的影响判断，字数控制在 1000 字以内，结论审慎不过度
- [ ] **授信报告章节完整性**：授信报告包含八个标准章节：授信申报情况、主体及业务简述、存量授用信、合规性审查、主体信用风险审查、授信业务审查、风险点及风控措施、审查结论意见
- [ ] **首段授信方案完整性**：第一章开头包含分行、客户、综合授信总额度、敞口额度、授信期限、授信品种、担保方式；授信品种不得缺失
- [ ] **申报性质标黄**：纯新增/存量续作/存量增额/存量减额等申报性质已用 `==文本==` 标黄，且与存量授信章节一致
- [ ] **审批权限标黄**：审批权限已标黄；若用户未确认总行级/部门级等具体级别，必须保留标黄待确认
- [ ] **存量授信处理**：纯新增业务写明“本次申报纯新增业务，无存量授用信”并标黄；存量业务按材料原文写明上轮批复和当前用信
- [ ] **征信查询标黄**：征信查询段落整体标黄；如无征信报告或数字不全，保留待补充提示
- [ ] **数据准确性**：授信金额、财务数字、担保金额、注册资本、股权比例、评级、征信余额等关键数字与数据摘要及关键来源摘录一致
- [ ] **来源注释完整性**：注释版需要保留的关键数据均带 `【来源：xxx】`；用户口述数据标注为客户经理口述或用户口述
- [ ] **清洁版可生成**：清洁版应去除 `【来源：xxx】` 注释，不残留来源标记
- [ ] **注释版可生成**：注释版应保留 `【来源：xxx】` 注释，文件名包含 `_注释版`
- [ ] **财务勾稽关系**：资产=负债+所有者权益；现金流、利润、资产负债表关键口径无明显矛盾，异常项已解释或标黄
- [ ] **财务分析针对行业**：财务分析结合行业和主营业务特征，不机械罗列指标；重点科目选择与客户行业匹配
- [ ] **重大异常处理**：应收、存货、负债、现金流、营收/成本等重大异常已有分析；材料无法解释的，已在报告中标黄并列入缺失/待确认项
- [ ] **缺失项处理**：无空白章节；材料不足的内容已写明“材料不足，待补充”或用户明确放弃，且待补充内容已标黄
- [ ] **还款来源审查**：第一还款来源、第二还款来源/担保措施均有分析，未使用“风险可控”“无信用风险”等绝对化表述
- [ ] **风险点与措施匹配**：风险点来自材料和分析结论，风控措施与风险点一一对应，不写泛泛空话
- [ ] **结论审慎合规**：审查结论与前文风险分析一致，未新增前文无依据的判断
- [ ] **模板符合度**：章节结构、标题层级、表格位置、固定表述符合 `references/report-template.md`
- [ ] **Word 格式符合度**：字体、字号、行距、缩进、页边距、表格边框、数字对齐符合 `references/format-guide.md`
- [ ] **文件命名规则**：清洁稿命名为 `{客户简称}_{YYYYMMDD}_授信审查报告.docx`，注释稿命名为 `{客户简称}_{YYYYMMDD}_授信审查报告_注释版.docx`
- [ ] **输出路径正确**：最终 Word 文件输出至当前约束指定的输出目录
- [ ] **原始材料安全**：未修改、移动或删除用户提供的原始材料

## 中间保存机制

每轮补充或修改后，自动保存进度：
- 保存路径：当前约束指定草稿目录下的 `<timestamp>_draft.md`
- 保存内容：当前报告 Markdown 内容 + 已确认的数据摘要 + 缺失项状态
- 用户可随时说"继续上次的报告"恢复进度

## 输出格式

### Word 文档规范
- 文件格式：`.docx`
- 字体：正文宋体小四，标题黑体（具体见 `references/format-guide.md`）
- 页边距、行距、页眉页脚等格式要求见 `references/format-guide.md`
- 使用 `scripts/generate_docx.py` 生成标准格式的 Word 文档

### 输出流程
1. 先向用户展示报告完整内容的 Markdown 预览
2. 用户确认或提出修改意见
3. 确认无误后，调用脚本生成 Word 文档
4. 告知用户文件保存路径
